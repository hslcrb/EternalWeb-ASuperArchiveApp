# EternalWeb 사용자 가이드 & 웹 아카이빙 통찰

이 문서는 EternalWeb의 사용법과 개발 철학, 그리고 현대 웹 아카이빙의 기술적 핵심을 담고 있습니다.

## 🌟 개발 철학: 왜 이터널웹(EternalWeb)인가?

인터넷은 인류 역사상 가장 방대한 지식의 저장소이지만, 동시에 가장 취약한 매체이기도 합니다. 오늘 본 중요한 신학적 자료나 뉴스 기사가 내일이면 '404 Not Found'로 사라질 수 있습니다.

EternalWeb은 단순히 파일을 다운로드하는 도구가 아닙니다. 사라져가는 **진실된 기록을 지키는 디지털 타임캡슐**이자, 후대에 전해줄 지식의 방패입니다.

---

## 🔍 웹 아카이브 vs 싱글파일: 무엇이 다른가?

보존 목적에 따라 최적의 도구를 선택해야 합니다.

| 구분 | 웹 아카이브 (Service/WARC) | 싱글파일 (SingleFile/HTML) |
| :--- | :--- | :--- |
| **저장 방식** | 브라우징 세션 및 네트워크 패킷 수집 | 현재 화면의 DOM 상태를 단일 HTML로 압축 |
| **주요 용도** | 사이트 박제, 동적 웹 기능 유지 | 개인 소장, 오프라인 학습 자료, 깔끔한 문서화 |
| **장점** | React, SPA 등 복잡한 사이트 재현율 높음 | 관리가 매우 쉽고 파일 하나로 모든 리소스 포함 |
| **단점** | 뷰어(Viewer)가 필요하며 파일이 무거울 수 있음 | 동적 API 요청 등 실시간 기능은 제한됨 |

---

## 🏗️ EternalWeb 3단계 통합 아키텍처

이터널웹은 위 두 방식의 장점만을 결합하여 **3단계 레벨 시스템**을 제공합니다.

### Level 1: 신속 보존 (SingleFile 엔진)
- 현재 페이지의 모든 이미지, 스타일, 폰트를 하나의 HTML 파일 안에 인라인(Base64)화하여 저장합니다.
- 복잡한 폴더 구조 없이 파일 하나만 복사하면 어디서든 원본과 동일한 디자인으로 볼 수 있습니다.

### Level 2: 상호작용 보존 (ArchiveWeb.page 엔진)
- 현대의 웹은 '생물'과 같습니다. 페이지를 연 후에도 자바스크립트가 서버와 통신하며 내용을 바꿉니다.
- Level 2는 사용자가 사이트를 돌아다니는 '과정'을 기록합니다. 덕분에 인터넷이 끊겨도 버튼을 누르면 다음 내용이 나오는 '살아있는' 아카이브를 만듭니다.

### Level 3: 심층 아카이브 (ArchiveBox 엔진)
- 한 페이지를 넘어 사이트 전체의 자산을 긁어모읍니다.
- PDF, 전체 스크린샷, 미디어 파일, 소스 코드 등 가능한 모든 형태로 동시 저장하여 데이터 소실 확률을 제로에 가깝게 만듭니다.

## ❓ 왜 아카이브가 깨지는 경우가 있나요?

현대 웹이 더 이상 단순한 '문서'가 아니라 **'실시간 소프트웨어'**이기 때문입니다.

1. **API 의존성**: 저장된 파일은 서버의 데이터베이스(DB)와 연결되어 있지 않습니다. "최신 댓글 가져와"라는 요청에 응답할 서버가 없으면 해당 영역은 비어 있게 됩니다.
2. **절대 경로 문제**: 일부 스크립트가 로컬 파일 시스템이 아닌 특정 URL에서만 작동하도록 강제되어 있으면 깨짐 현상이 발생합니다.
3. **보안 벽**: 로그인 세션이나 유료 결제 장벽 뒤에 있는 콘텐츠는 수집기가 접근할 수 없습니다.

---

## 🚀 사용 팁 (Best Practices)

1. **중요한 텍스트 자료**: SingleFile(Level 1)로 저장하되, '스크립트 제거' 옵션을 사용하면 10년 뒤에도 가장 안전하게 읽을 수 있습니다.
2. **복잡한 웹앱/커뮤니티**: ArchiveWeb.page(Level 2)로 직접 클릭하며 기록하세요.
3. **영구 증거 수집**: ArchiveBox(Level 3)를 통해 PDF와 스크린샷을 동시에 확보하여 공신력을 높이세요.
